# N¨UWA: MENDING THE SPATIAL INTEGRITY TORN BY VLM TOKEN PRUNING 论文总结

## 1. 研究动机
- 视觉语言模型(VLM)在推理过程中生成大量视觉token，导致显著的计算开销和吞吐量降低
- 现有视觉token剪枝方法在视觉问答(VQA)任务上表现出色，但在视觉定位(VG)任务上性能严重下降
- 作者通过分析发现，基于全局语义相似性和注意力分数的剪枝策略会丢失全局空间参考框架，该框架源于token位置信息的交互
- 这一限制阻碍了VLM在需要精确空间定位的任务中的应用，亟需解决任务间性能不一致的问题

## 2. 解决问题
- 如何在视觉token剪枝过程中保持空间完整性，特别是对于需要精确空间定位的VG任务
- 解决现有剪枝方法在VQA和VG任务上表现显著不一致的问题
- 在减少token数量的同时保留全局空间参考框架，这对空间感知至关重要

## 3. 现象分析
- 系统评估表明，先进剪枝方法在VQA任务上与简单基线相比优势有限，但在VG任务上所有方法都表现出系统性退化
- 平均池化方法在剪枝方法中表现最佳，可能因为它部分保留了空间结构特征
- 通过注意力流分析和梯度加权归因方法，揭示了VLMs中的两阶段视觉处理流程：
  - 早期阶段：任务无关的多模态交互
  - 中期阶段：任务特定的处理，VG任务在此阶段对视觉token有更高依赖
- 视觉注意力熵(VAE)和对象中心凝聚性(OCC)指标显示，视觉处理在ViT编码器和LLM解码器的中期阶段形成对象级别表示
- 位置重建实验证实，VG任务性能下降的主要原因是token剪枝策略中丢失了全局空间参考框架

## 4. 主要方法
### N¨uwa框架：两阶段空间感知token剪枝

#### 第一阶段：视觉编码器中的空间凝聚剪枝
1. **分离(Separation)**：
   - 将token图划分为M×M非重叠局部区域
   - 确保完整的全局坐标系统

2. **对齐(Alignment)**：
   - 在每个区域内选择具有高全局显著性的基准token
   - 使用显著度分数选择：S(t_i) = α_cls,i × ||k_i||_2
   - α_cls,i是[CLS]token的注意力权重，||k_i||_2是token的key向量的L2范数（信息容量）

3. **聚合(Aggregation)**：
   - 基于空间邻近性合并token特征
   - 角色分配：高容量token为支柱token(T_P)，特征保持不变；其余为收集者token(T_C)，从空间邻居聚合特征
   - 加权聚合：结合语义相似矩阵(A)和空间邻近矩阵(P)形成权重矩阵W

#### 第二阶段：LLM中的文本引导剪枝
- 在LLM多模态特征对齐后的中间层执行
- 利用文本语义指导进一步的剪枝
- 保留与任务相关的视觉token

## 5. 数据与实验
### 数据集
- **VQA基准**：GQA、MMB、MMMU、MME、VQAv2、VQAtext、POPE、SQA、MMVet
- **视觉定位基准**：RefCOCO系列(Refcoco-test、Refcoco+-testA、Refcoco+-testB、Refcocog-test等)

### 基线模型对比
- **Vision Encoder-Side Pruning**：VisionZip、PruMerge+
- **LLM Single-Layer Pruning**：FastV
- **LLM Multi-Layer Pruning**：PyramidDrop(PDrop)、SparseVLM
- **简单基线**：随机采样、平均池化

### 核心实验结果
- **VQA任务**：实现SOTA性能，从94%提升到95%
- **VG任务**：实现显著改进，从7%提升到47%
- 在88.9%的视觉token减少下：
  - VQA任务保持95%性能
  - VG任务保持47%性能
- 实现了89%的TFLOPs减少和62%的prefill时间减少
- 在13个数据集和多个VLM上验证了方法的有效性

## 6. 主要贡献
1. **任务特定分析**：
   - 系统检查VLM处理流程，揭示当前剪枝方法因忽略任务特定需求和破坏空间结构而在定位任务上失败
   - 位置重建实验证实空间感知源于全局参考框架的完整性

2. **N¨uwa框架**：
   - 提出两阶段空间感知剪枝框架，通过分离和自适应区域聚合保留全局空间锚点
   - 保持空间和语义完整性，同时利用LLM中的文本信息进行基于多模态对齐的剪枝

3. **性能验证**：
   - 在13个数据集和多个VLM上取得优越结果
   - 在VQA任务上实现95%性能保留，VG任务上实现47.2%性能保留，建立新的SOTA
   - 在88.9%的token减少下实现89%的TFLOPs减少和62%的prefill时间减少
